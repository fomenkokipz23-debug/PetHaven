# PetHaven - Система керування готелем для тварин

**PetHaven** - це консольний застосунок для автоматизації процесів обліку, заселення та контролю номерного фонду в готелях для домашніх улюбленців. Проєкт розроблено в межах вивчення об'єктно-орієнтованого програмування, SOLID-принципів та архітектурних шаблонів на платформі .NET.

## 📌 Архітектура проєкту (Clean Architecture)

Застосунок розділено на ізольовані шари (проєкти) для забезпечення гнучкості розширення та незалежності від технологій збереження даних:

* **`PetHaven.Domain`** - ядро системи. Містить сутності предметної області (`Pet`, `Room`, `Booking`), енуми, інтерфейси репозиторіїв та локальні бізнес-правила (захист інваріантів об'єктів).
* **`PetHaven.Application`** - шар бізнес-логіки. Містить сервіси (`BookingService`, `AnalyticsService`), які керують сценаріями використання системи та виконують LINQ-аналітику.
* **`PetHaven.Infrastructure`** - технічна реалізація. На другій ітерації додано довготривале **асинхронне файлове збереження даних у форматі JSON** (`JsonFileStore<T>`), що замінило базове збереження в пам'яті.
* **`PetHaven.Console`** - текстовий інтерфейс користувача (UI), що забезпечує взаємодію адміністратора готелю з бізнес-сервісами через розширене меню.
* **`PetHaven.Tests`** - проєкт з юніт-тестами (xUnit) для автоматичної перевірки стабільності доменної моделі, бізнес-правил та патернів розширення.


## 🚀 Функціонал Ітерації 2 (Лабораторна робота №35)

У межах другої ітерації застосунок перейшов від демонстраційного стану до повноцінного прикладного рішення:

1. **Довготривале збереження (Persistence):** Реалізовано асинхронні методи `LoadAsync` та `SaveAsync` з використанням `System.Text.Json` для автоматичного відновлення стану готелю при старті програми та збереження змін у файли `rooms.json` та `bookings.json`.
2. **Розширення бізнес-логіки (3 Use Cases):**
   * *UC-1: Заселення тварини (Check-In)* з динамічним розрахунком вартості.
   * *UC-2: Виселення тварини (Check-Out)* зі зміною статусу бронювання та автоматичним звільненням кімнати.
   * *UC-3: Скасування бронювання* з валідацією поточного стану.
3. **Механізм розширення (Патерн Strategy):** Впроваджено інтерфейс `IPricingStrategy` для розрахунку вартості проживання. Реалізовано дві стратегії: `RegularPricingStrategy` (стандартна) та `AnimalSpecificPricingStrategy` (з урахуванням особливостей догляду: знижка для котів, націнка для собак).
4. **LINQ-аналітика:** Впроваджено `AnalyticsService` для формування звітів:
   * Фільтрація активних постояльців.
   * Підрахунок загального прибутку готелю (`Sum`).
   * Групування та підрахунок тварин за типами з використанням спеціалізованої колекції `Dictionary<PetType, int>`.
   * Пошук топ-найдовших бронювань (`OrderByDescending`, `Take`).

## Контрольні питання (Лабораторна робота №35)

### 1. Чому persistence-шар повинен залежати від контрактів, а не навпаки?
 Це головна вимога принципу **Dependency Inversion (D із SOLID)** та Чистої архітектури. Шари бізнес-логіки (`Domain` та `Application`) визначають правила гри у вигляді інтерфейсів (контрактів), наприклад `IBookingRepository`. Шар збереження даних (`Infrastructure`) є лише технічною деталлю. 

Якщо інфраструктура залежить від доменних контрактів, ми можемо безболісно замінити JSON-файли на базу даних PostgreSQL, XML або хмарне сховище, абсолютно не змінюючи і не перекомпілюючи код нашої бізнес-логіки. Якщо зробити навпаки - бізнес-логіка стане "заручником" конкретної технології чи структури файлів.

### 2. Які з ваших бізнес-правил належать сутностям, а які сервісам?
* **Сутностям (Domain Entities)** належать внутрішні інваріанти об'єкта, які він може перевірити самостійно без сторонньої допомоги. Об'єкт принципово не може увійти в некоректний стан. *Приклади в нашому проєкті:* валідація віку тварини (>0) у `Pet`, заборона створення бронювання з датою виїзду, що передує заїзду в `Booking`, а також правило, що завершити чи скасувати можна лише активне проживання.
* **Сервісам (Application Services)** належать оркестраційні або глобальні правила процесу, які потребують залучення інших систем чи даних з репозиторіїв. *Приклади в нашому проєкті:* перевірка у `BookingService`, чи взагалі існує кімната з таким номером, та перевірка, чи не зайнята вона прямо зараз іншим улюбленцем (сутність `Booking` сама по собі не має доступу до списку всіх кімнат у файлах, це знає лише репозиторій).

### 3. Чому LINQ-запити мають бути частиною бізнес-сценарію, а не окремою “лабораторною прикрасою”?
У реальних корпоративних системах LINQ є основним інструментом для виконання бізнес-аналітики, фільтрації даних та ухвалення рішень на рівні застосунку. 

У нашому проєкті LINQ-запити вирішують реальні бізнес-завдання адміністратора: розрахунок фінансового доходу готелю (`Sum`), оперативний моніторинг завантаженості готелю (`Where`), оптимізація закупівель корму на основі підрахунку кількості котів/собак (`GroupBy` + `ToDictionary`). Вони інтегровані в `AnalyticsService` як частина сценаріїв звітності, а не просто викликаються в порожнечу для демонстрації синтаксису.

### 4. Як обраний патерн проєктування допоміг уникнути жорсткого кодування поведінки?
Обраний патерн **Strategy** (`IPricingStrategy`) повністю усунув жорстке кодування (hardcoding) алгоритму розрахунку цін всередині класу `Booking`. Замість використання заплутаних конструкцій `if/else` чи `switch` прямо в доменній моделі, сутність делегує розрахунок зовнішній стратегії.

**Вигода:** Якщо готель захоче ввести новий тариф (наприклад, "Новорічний +20%" або "Для постійних клієнтів -15%"), нам достатньо буде просто створити новий клас, який реалізує `IPricingStrategy`. Код сутності `Booking` та сервісу `BookingService` при цьому залишиться абсолютно недоторканим, що повністю відповідає принципу **Open/Closed (O в SOLID)**.

### 5. Що саме Lab 36 повинна довести про ваш код після цієї ітерації?
Лабораторна робота №36 присвячена тестуванню (Quality Gate та Fault Handling). Вона повинна довести:
1. **Надійність Persistence-шару під навантаженням або в екстремальних умовах:** Як поведеться система, якщо файл JSON буде пошкоджено, заблоковано іншим процесом, або якщо він взагалі зникне з диска під час роботи програми.
2. **Коректність ізоляції через Mock-тести:** За допомогою інтеграційних та розширених тестів ми повинні довести, що наші сервіси (`BookingService`) працюють правильно незалежно від реальних файлів на диску, імітуючи поведінку репозиторіїв за допомогою заглушок.
3. **Стійкість до регресій:** Що додавання нових фіч не ламає логіку, яку ми заклали та протестували на Ітераціях 1 та 2.
